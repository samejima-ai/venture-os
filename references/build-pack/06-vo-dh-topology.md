# VO↔DH 連携トポロジー ── 二重入れ子ループ

> ビルド準備パック #6（連携）。VO-L1 が DH をどう呼ぶかの契約。L1 SKILL の spec に効く。
> 根拠: CONCEPTION v2 §2（D1 連携契約）/ philosophy 第1条フラクタル / 憲法 V3・V4。

## 中核: DH は VO-L1 の内側で呼ぶ条件付き・反復の下請け

VO と DH は直列ではなく **入れ子**。VO の事業ループ（外）が回り、必要な時に DH のソフト開発ループ（内）を発火させて閉じる。両方とも A⇄B 擦り合せループ（第1条）で同一形状。

```
VO-L0（事業の対話設計・D4 汎用設計FW）
  └→ 事業執行環境を生成（THESIS/VENTURE-REGIME/DONT-VENTURE/信用・貨幣センサー/Council/monetary_gate）
       └→ VO-L1 実働 ＝【外ループ】信用獲得 ⇄ 充足マネタイズ（V3 代謝ループ・常時回転）
             │
             ├─（ソフトが要る時だけ・反復）→ DH-L0（THESIS→SPEC の橋）
             │        └→ 開発環境生成 → DH-L1 ＝【内ループ】実装 ⇄ 自己検証
             │              └→ 献上＝プロダクト ──┐
             │                                    ↓（VO-L1 の信用獲得/マネタイズの"手段"として戻る）
             │←──────────────────────────────────┘
             │
             └─（信用毀損/資金枯渇＝運用インシデント）→ VO-L0 へ差し戻し（V4・pivot・L3非新設）
```

## DH 呼び出しの3性質

| 性質 | 内容 | L1 spec への含意 |
|---|---|---|
| **条件付き** | ソフトが要る事業だけ DH を呼ぶ。design_only / 非ソフト事業は DH を一度も呼ばない | VO-L1 に「ソフト必要性の判定」ステップを持つ |
| **入れ子** | DH は VO-L1 実働の内側で起動（"プロダクトが要る"と判明した時） | VO-L1 → DH-L0 の呼び出し契約（THESIS→SPEC 変換） |
| **反復** | 事業の生涯で何度も DH サイクルが走る（機能追加のたび） | 単発でなく複数 DH サイクルを束ねる管理 |

## 連携契約（第3条 情報純度の入出力規格）

| 方向 | 渡すもの | 変換 |
|---|---|---|
| VO → DH | `THESIS.md`（顧客×課題×価値提案×信用ポジション×BMパターン×ユニットエコノミクス前提） | 事業テーゼ → DH `SPEC.md` の WHY 層＋機能要求 |
| DH → VO | プロダクト＋計算可能代理指標（CVR/完了率/レイテンシ/エラー率） | ソフトの成果 → VO-L1 の信用・貨幣センサー入力 |

## 実例（ai-gata が経験的証拠）

ai-gata は「（暗黙の）VO が DH を繰り返し発注して作られた事業」。F1（診断エンジン）→ F6（OGP）→ F15（次の一歩）… と **DH サイクルが反復発火**した履歴そのものが、この入れ子トポロジーの実証。VO が明示化されていれば、起点に VO-L0（AI型診断事業の信用戦略・マネタイズ設計）が置かれ、各機能が DH 発注として走った。

## 二重ループの単位対応（第1条フラクタル）

| | 外ループ（VO） | 内ループ（DH） |
|---|---|---|
| A⇄B | 信用獲得 ⇄ 市場検証 | 実装 ⇄ 自己検証 |
| 素材 | 事業（信用・貨幣） | ソフト（コード） |
| 勝利条件 | 継続（V4） | 仕様合致・動作 |
| インシデント帰着 | VO-L0（テーゼ再策定） | DH-L0（仕様差し戻し） |

両ループは入れ子でも同一形状 ── フラクタル原則の綺麗な現れ。
