本書は「DH（dialog-harness）メタスキルの兄弟メタスキルとしての venture-os」の考案を、DH の**フラクタル原則**（第1条）に忠実に再構築したものである。実装ではなく設計思想の献上物であり、凍結事項の確定・スコープ確定は人間の判断（P1発案 / H3 方向性発案）に委ねる。

## 0. 一行で言うと

**DH は「人間の意図 → 自律的なソフトウェア開発」を回す OS。venture-os はその双対で「人間の意図 → 自律的な事業の設計・検証・運営」を回す OS。**

両者は同じ深層形状（A⇄B 擦り合せループ / Shift Left / 情報純度 / 献上 / Council / 委譲境界）を共有し、**素材だけが違う**。DH の素材は *ビット（コード）*、venture-os の素材は *事業（ビジネスモデルと市場）*。

この「同一形状・異素材」という関係そのものが、DH 第1条フラクタル原則の最上位インスタンスである。DH と venture-os は階層的な親子ではなく、**直交する兄弟**（philosophy.md 第1条「L0/L1/L2 は階層ではなく直交」の一段上への適用）。

## 1. なぜ「兄弟」であって「応用」でないのか

DH を事業ドメインに「応用」するだけなら、新メタスキルは要らない（DH の L1 に事業計画を書かせればよい）。兄弟として独立させる根拠は、**素材の性質が一点で決定的に異なり、その差が憲法の重心を移動させる**からである。

### 決定的差分＝可逆性（第9条の重心が逆）

| 観点 | DH（コード） | venture-os（事業） |
|---|---|---|
| 標準操作の可逆性 | **高い**（revert / amend / squash で原状回復） | **低い**（資金投下・法人設立・雇用・契約・市場での評判は revert 不能） |
| ゆえに委譲の既定重心 | **L-FULL 全自律**（出力後修正モデル・流速優先） | **L-GATE 事前ゲート寄り**（不可逆行為が多数派） |
| 「9割は推奨でいい」 | 可逆領域なので成立 | 可逆領域（コピー・価格 A/B・LP 文言）に**限って**成立 |

DH 第9条は「原状回復できるか *だけ* で線を引く」と宣言した。この判定軸をそのまま venture-os に持ち込むと、**同じ原則が正反対の運用を導く**。コードでは revert 可能ゆえ大半が L-FULL に落ちるが、事業では不可逆行為（金・法・人）が多数派ゆえ大半が L-GATE / L-FROZEN に上がる。

> これは憲法の書き換えではなく、**同一憲法（第9条）の異素材への正しい適用**である。ここにこそ兄弟を分ける必然性がある。DH の crosscut-issue-implementer が「ready-for-ai ラベル = 人間承認」を経てのみ実装を起動するのと同型に、venture-os は「不可逆行為の実行前ゲート」を構造的に多く持つ。

### 素材が変えるもの / 変えないもの

- **変えない（兄弟が共有する不変核）**: 9条憲法の *形状*（A⇄B ループ、Shift Left、情報純度、献上、Council、可逆性ベースの委譲、L3 を新設しない）。
- **変える（兄弟で異なる素材依存の充填）**: 正典（コード規約 → ビジネスモデル大辞典）、SPEC の中身（機能仕様 → 事業仮説）、センサー（型/lint/test → ユニットエコノミクス/市場テスト）、Council ペルソナ（経営者/開発者/哲学者 → 投資家/創業者/懐疑的顧客）、委譲境界の既定値（可逆判定の結果が逆）。

## 2. 正典（Canon）── 大辞典が venture-os の憲法下位規範になる

DH では正典が二層ある：(a) 全プロジェクト共通の `philosophy.md`（9条憲法）と、(b) プロジェクト固有の凍結コンテンツ（ai-gata なら診断コンテンツ 16タイプ・8要素）。

venture-os も同型に二層：

| 層 | DH | venture-os |
|---|---|---|
| 共通憲法 | `philosophy.md`（9条） | `philosophy.md` を継承 + venture 差分注記（本書 §4） |
| 共通正典（設計語彙） | ARC パターン3種（monolith / realtime-pubsub / event-sourcing） | **ビジネスモデル大辞典 66パターン（A〜G 7分類）** |
| プロジェクト固有凍結 | 各プロダクトの凍結コンテンツ | 各事業の凍結テーゼ（顧客・課題・価値提案の核） |

### 大辞典＝ARC パターンライブラリの事業版（本書の中核主張）

DH の L0（spec-architect）は REGIME 判定時に **ARC パターンを1つ選ぶ**（monolith が既定、NFR スコアで切替）。venture-os の L0 は同じ位置で **BM パターンを選ぶ／組み合わせる**。大辞典はそのための選択メニューであり、venture-os にとって「アーキテクチャ選択肢の凍結ライブラリ」に相当する。

この写像が成立する根拠は大辞典自身が持つ2つの構造にある：

1. **A〜G の7分類**（課金 / プラットフォーム / 流通 / 資産オペ / 価値提案 / 財務 / 顧客市場選択）= venture-os の設計軸そのもの。DH の ARC が「デプロイ構造の選択」なら、A〜G は「収益構造の選択」。
2. **St. Gallen マジックトライアングルの Who/What/How/Value タグ**（大辞典 §第IV部）= DH の REGIME スコアリング軸に相当する**直交タグ体系**。大辞典は「A〜F のテーマ分類と直交するので二軸併記が可能」と明記する。これは venture-os の REGIME.md が「BM パターン（テーマ軸）× Who/What/How/Value（影響次元軸）」の二軸で事業を記述できることを意味する。

### 「90%は再結合」テーゼ ＝ venture-os の設計方法論

大辞典は St. Gallen の「新規 BM の約90%は既存パターンの再結合」テーゼを引く（帰属付き・未監査推計として）。これは DH の設計思想と正確に対応する：DH が ARC + 機能パターンの**組み合わせ**でシステムを構成するように、venture-os は BM パターンの**組み合わせ**で事業を構成する。第III部の AI 時代モデル（埋め込み型金融 = レイヤープレイヤー × Object-as-PoS × 替刃 の再結合、等）は、まさに「パターン合成による新規事業設計」の実例カタログとして機能する。

→ **venture-os の L0 の中核タスクは「白紙から事業を発明する」のではなく「大辞典パターンの適切な再結合を人間と擦り合せる」こと。** これは DH の L0 が「白紙からアーキテクチャを発明せず、ARC + 既知パターンを選ぶ」のと同型。

### 確度メタデータの継承（凍結境界の運用）

大辞典は各主張に検証ステータス（`検証3-0` 全会一致 / `未検証・出典付き` / `反証棄却`）を付す。これは DH の献上物「確度」タグ（`確度: 人間確定` / `AI推定` ── INTENT.md の運用）と同型である。venture-os では：

- **凍結**: `検証3-0` 相当の確定パターン（A〜G の66パターン核定義）。変更は人間へ献上（無断改変禁止 ＝ ai-gata CLAUDE.md「凍結境界を守る」の venture 版）。
- **未凍結・要検証**: 第III部の AI 時代候補（敵対的検証未通過）。venture-os の L1 が市場検証で昇格/棄却する対象。
- **反証棄却**: 使用禁止ソース（exploit-advisory.ch 由来2件）＝ DONT 相当。

## 4. 9条憲法の翻訳（何がそのまま・何が変異するか）

DH 憲法は**そのまま継承**が原則。以下は各条の venture 素材への適用注記のみ（条文の書き換えではない）。

- **第1条 フラクタル**: そのまま。A⇄B ループが「人間⇄L0（テーゼ擦り合せ）/ L1⇄市場（仮説⇄検証）/ L2⇄L1群 / L1⇄独立検証者」で反復。**L3（恒常運用層）を新設しない**＝事業運営インシデント（解約増・CAC 悪化）は「新テーゼの発見」として L0 に戻す。
- **第2条 Shift Left**: そのまま。ただし「決定論的に解ける問題」の中身が**算数**になる。CAC>LTV が算数で自明な事業は市場テスト前に棄却（推論・実市場投下の前に計算で潰す）。「E2E の前に Console 根絶」＝「顧客インタビューの前に財務モデルの破綻を根絶」。
- **第3条 情報純度**: そのまま。独立市場検証者を立てる4基準（純度向上/責務明確/入出力規格化/独立コンテキスト必然性）は「創業者の確証バイアスから隔離して仮説を反証する」必然性で完全に満たす。
- **第4条 人間責務**: そのまま。人間＝意図・スコープ・不可逆判断の承認 / AI＝ドキュメント化・アセット生成・検証。「UX は計算可能代理指標まで」＝「事業性は計算可能代理指標（コンバージョン率・CAC・LTV・粗利）まで。創造的ブランド *感性* は射程外」。
- **第5条 献上**: そのまま。A/B/C/D の4タイプを事業版に写像（§6）。
- **第6条 人間≒Council**: そのまま。**H カテゴリ（人間専管）が事業では拡大**する ── 資金調達・法人設立・重要契約・撤退判断は H4 相当（根本設計見直し）。VTL による委譲でも金・法・人は残す。
- **第7条 AI組織論**: そのまま。4役割（L0設計/L1実装/L2統括/Council判断）+ サポート。P1〜P4（発案/ブレスト/事後確認/暴走介入）も同型。**P4 暴走介入＝バーンレート急増・法務リスク検知時の circuit breaker**。
- **第8条 自律性（観測→候補化→人間承認）**: そのまま。事業では「候補化」が「実行推奨」に、「人間承認」が「予算・GO 承認」になる。CTL 0 で候補出力すら抑止＝コールドスタート事業では AI は観測のみ。
- **第9条 委譲境界（可逆性）**: **原則継承・重心逆転**（§1）。venture-os 固有の L-GATE 領域: 資金投下・広告予算執行・法人/契約/雇用・価格の対外恒久変更・撤退。L-FULL 領域: LP コピー・価格 A/B（可逆な範囲）・市場調査・財務モデル・GTM プラン草案。

> **憲法の自己改訂禁止（第6条・第9条 L-FROZEN-META）はそのまま最優先で継承**。venture-os も自らの委譲境界 SPEC を自ら改訂できない。

## 5. 5層 事業検証スタック（第2条の事業版）

DH の5層検出スタック（計算的→E2E→Interaction Cost→推論→人間）を事業に写像。左ほど安価・先行。

| 層 | DH | venture-os | 判定するもの |
|---|---|---|---|
| 基盤（設計時予防・30%） | 型/状態機械/不変条件 | **ユニットエコノミクス設計**（LTV/CAC/粗利/回収期間の式を先に置く） | 算数で破綻する事業を発生前に防ぐ |
| 第1層 計算的（30%） | build/type/lint/test | **財務モデル整合**（TAM 上限・損益分岐・runway 計算） | 決定論で解ける破綻（負の粗利・CAC>LTV） |
| 第2層 機械検証（20%） | Playwright E2E | **市場スモークテスト**（LP + 広告で実クリック/CVR 実測・スマートテスト） | 需要の実在（誰かが実際に反応するか） |
| 第3層 代理指標（10%） | Interaction Cost | **獲得コスト代理指標**（CAC/CVR/CTR/オンボ完了率の定量閾値） | 効率が閾値内か |
| 第4層 推論（7%） | LLM 仕様合致判定 | **市場適合の推論判定**（顧客インタビュー要約・競合分析の LLM 判定） | 定性シグナルの構造化 |
| 第5層 人間（3%） | Vision / 人間 | 人間の事業勘・撤退/継続判断 | 数値化不能な最終判断 |

**要点**: DH 第2条「推論の前に計算」がここでも効く。市場に金を投じて検証する（第2層）前に、財務モデル（基盤・第1層）で自明な破綻を潰す。これが「Shift Left の事業版」であり、venture-os が単なる「AI 事業計画ジェネレータ」と一線を画す構造的理由。

## 6. 献上4タイプの事業版（第5条）

| タイプ | DH | venture-os | 開発継続 |
|---|---|---|---|
| A 仕様レビュー結果 | 実装不可能で差し戻し | **pivot 要求**（テーゼが算数/市場で棄却・L0 でテーゼ再策定） | 中断 |
| B 成果物 | 動くコード + 検証 | **検証済み BM**（市場スモーク通過の事業アセット一式 + ユニットエコノミクス実測） | 完了 |
| C 仕様改訂提案 | 実装続行可・改善提案 | **仮説改訂提案**（実行中に発見した BM パターン差替・価格改訂の提案。献上時一括） | 継続 |
| D 異常献上 | 技術的例外 | **市場・規制・資本の実行不能**（規制障壁・資金枯渇・チャネル遮断。テーゼは正しいが実行不能） | 中断 |

タイプ A（テーゼ起因の pivot）とタイプ D（実行手段起因の障害）を分離するのは DH と同じ純度原則。「BM が悪い（A）」と「BM は良いが規制で無理（D）」を混同しない。

## 7. venture_scope 軸（第8条・段階的自律の事業版）

DH の dev_mode（local_only → github_assisted → autonomous）に対応する、**外部露出と自律実行の段階**。スコープの三択を1つの軸に溶かす（DH 流の「フォークをスペクトラムにする」設計）。

| venture_scope | 出力 | 外部への金/評判の露出 | 人間関与 |
|---|---|---|---|
| **design_only**（既定・最保守） | テーゼ + BM 設計 + 財務モデル + GTM プラン（紙まで） | なし（可逆） | L0 承認のみ |
| **gtm_execution** | 上記 + LP 公開・小額広告・価格 A/B・MVP 定義 | 小（予算ゲート付き） | 予算 GO + P4 監視 |
| **full_ops** | 上記 + 継続運営・自律実験・指標監視ループ | 大 | P1〜P4 + 不可逆ゲート |

DH の autonomous_scope（full/merge_gated/custom）と同型に、既定は**保守側**（design_only）。DH が「GitHub 無しでも完全動作」を原則とするように、venture-os は「**実市場に出さなくても（design_only で）完全動作**」を原則とする ── 設計と検証（机上）だけで価値を出せる。

## 8. レイヤ構造とスキル配置案（実装したらこうなる）

DH のスキルツリーに対応する venture-os スキル群の初期案（**考案段階・確定でない**）：

```
L0 設計   layer0-venture-architect     テーゼ策定・BMパターン選択・VENTURE-REGIME 判定
          layer0-venture-onboarding    既存事業の逆テーゼ化（走っている事業を harness 化）
          layer0-venture-reindex       運営履歴（episodic）→ 事業叡智へ再蒸留
L1 実行   layer1-venture-exec          thesis→市場アセット生成・自己検証
          layer1-market-reviewer       独立市場検証（反証専門・実行バイアス隔離）

```
